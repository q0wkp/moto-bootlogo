## 1. 文件格式分析

### 1.1 整体结构

Motorola boot logo 分区采用 **MotoLogo** 容器格式，结构如下：

```
┌─────────────────────────────────────────────────────────────┐
│ [0x000-0x008] Magic: "MotoLogo\0" (9 bytes)                 │
├─────────────────────────────────────────────────────────────┤
│ [0x009-0x00C] 索引表结束位置 (4 bytes, little-endian)        │
│               值 = 0x3AD，计算图像数量: (0x3AD - 0x0D) / 0x20 = 29 │
├─────────────────────────────────────────────────────────────┤
│ [0x00D-0x3AC] 图像索引表 (29 × 32 bytes = 928 bytes)        │
│               每条目:                                        │
│               ├── 名称 (24 bytes, null 填充)                 │
│               ├── 偏移量 (4 bytes, little-endian)            │
│               └── 大小 (4 bytes, little-endian)              │
├─────────────────────────────────────────────────────────────┤
│ [0x3AD-0x3B0] 版本号: 0xFFFFFFFF = -1 (基础版本)             │
├─────────────────────────────────────────────────────────────┤
│ [0x3B1-0x3FF] 填充 (0xFF)                                   │
├─────────────────────────────────────────────────────────────┤
│ [0x400-0xE32FDF] 图像数据区 (每张图按 0x200 对齐)            │
├─────────────────────────────────────────────────────────────┤
│ [0xE32FE0-0xE330FF] 签名区域 (288 bytes)                    │
│                     ├── "PROD:avatrn     " (16 bytes)       │
│                     ├── "CSV:15          " (16 bytes)       │
│                     └── 加密签名 (256 bytes)                 │
├─────────────────────────────────────────────────────────────┤
│ [0xE33100-0x1FFFFFF] 零填充至 32MB                          │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 单张图像格式 (MotoRun RLE)

每张图像使用 **MotoRun** 格式，采用 RLE (Run-Length Encoding) 压缩：

```
┌─────────────────────────────────────────────────────────────┐
│ [+0x00] Magic: "MotoRun\0" (8 bytes)                        │
├─────────────────────────────────────────────────────────────┤
│ [+0x08] 宽度 (2 bytes, big-endian)                          │
│ [+0x0A] 高度 (2 bytes, big-endian)                          │
├─────────────────────────────────────────────────────────────┤
│ [+0x0C] RLE 压缩像素数据                                     │
└─────────────────────────────────────────────────────────────┘
```

**RLE 编码规则**：
- 每个数据块以 2 bytes 开头 (big-endian)
- 最高位 (bit 15) 为重复标志：
  - **1**: 后面 3 bytes 是颜色 (BGR)，重复 N 次
  - **0**: 后面 N×3 bytes 是连续不同的像素
- 颜色顺序是 **BGR** 而非 RGB

## 2. 现有解析工具 moto-bootlogo.py 分析

这是一个 Python 脚本，功能：
- **解包**: 读取 MotoLogo 文件，解码 RLE，输出 PNG 图像
- **打包**: 读取 PNG 图像，编码 RLE，生成 MotoLogo 文件

**发现的问题**：

1. **MD5 去重逻辑**
   - 脚本对每张图像计算 MD5，如果相同则共享偏移量
   - 导致修改后的 orange1/orange2 错误指向 logo_boot 的数据
   - 原始偏移量被覆盖，设备读取时解码错误

2. **缺少尾部填充**
   - 打包后文件约 14MB，而原始文件是 32MB
   - 脚本没有填充到分区大小

3. **签名区域丢失**
   - 打包时没有保留原始文件尾部的签名数据

## 3. 实验过程

### 3.1 实验一：直接使用 moto-bootlogo.py 打包

**操作**：
1. 解包 `logo_a.bin` 到 `logo_unpacked/`
2. 不做任何修改，直接打包回 `logo_fixed.bin`

**结果**：
- 输出文件只有 14MB
- orange1/orange2 被去重，指向 logo_boot 的偏移量
- 刷入后 orange1/orange2 显示黄字错误

**分析**：
```
原始 orange1: offset=0x8E8600, size=343942
打包后:       offset=0x400,    size=5933169  ← 错误指向 logo_boot!
```

### 3.2 实验二：修复去重和填充问题

**操作**：
1. 修改脚本，添加 `--no-dedup` 禁用去重
2. 添加 `--pad 33554432` 填充到 32MB
3. 重新打包

**结果**：
- 文件大小正确 (32MB)
- orange1/orange2 有独立偏移量
- 仍然显示黄字错误

**分析**：文件末尾缺少签名数据。

### 3.3 实验三：保留签名区域

**操作**：
1. 创建 patch 工具，只替换指定图像
2. 复制原始文件的签名区域到输出文件
3. 用纯黑色图像替换 orange1/orange2

**结果**：❌ 仍然显示黄字错误

### 3.4 实验四：只修改索引表（不重新编码）

**操作**：
1. 不重新编码任何图像
2. 只修改 header 中 orange1/orange2 的 offset 字段
3. 让它们指向 yellow1/yellow2 的原始数据

**结果**：❌ 仍然显示黄字错误

### 3.5 实验五：原位覆盖数据（header 完全不变）

**操作**：
1. Header 中的 offset 和 size 完全保持原样
2. 只在 orange1 的原始 offset 位置覆盖新的 RLE 数据
3. 用 0x00 填充剩余空间

**结果**：❌ 仍然显示黄字错误

### 3.6 实验六：测试 logo_boot 是否可修改

**操作**：
1. 用纯红色图像替换 logo_boot
2. 其他图像保持原始数据

**结果**：✅ **成功！开机显示红屏**

## 5. 结论

### 5.1 选择性签名验证

Motorola Edge 2024 的 bootloader 实现了**选择性签名验证**：

|              图像类型              | 签名验证 |   可修改性   |
| --------------------------------- | ------- | ---------- |
| 普通图像 (logo_boot 等)             | ❌ 无   | ✅ 可修改   |
| 安全警告图像 (orange/yellow/red 等) | ✅ 有   | ❌ 不可修改 |

### 5.2 受保护图像（推测）

根据命名规律，以下图像可能受签名保护：
- `orange1`, `orange2` — Bootloader 解锁橙色警告
- `yellow1`, `yellow2` — 黄色警告状态
- `red1`, `redeio1` — 红色警告状态  
- `lock_cfm`, `lock_cfm2` — 锁定确认画面
- `unlock_cfm`, `unlock_cfm2` — 解锁确认画面

### 5.3 签名验证机制推测

签名可能不是针对整个文件计算的，而是：
- 每张受保护图像单独计算签名，或
- 受保护图像的数据区域有独立的校验值

无论如何，签名密钥由 Motorola 持有，无法伪造。

### 5.4 技术限制

1. **签名算法未知** — 可能是 RSA/ECDSA
2. **私钥不可获取** — Motorola 专有
3. **无已知绕过方法** — 设备于 2024 年 6 月发布，较新

## 6. 结论

**orange1/orange2 无法通过软件方式修改或隐藏**。这是 Motorola 的安全设计，用于确保用户在 bootloader 解锁后始终能看到安全警告。
